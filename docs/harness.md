# 하네스

## 1. 문서 목적
이 문서는 미팅 에이전트 프로젝트에서 AI가 따라야 할 작업 범위, 경계, 제약, 산출물 기준을 정의하기 위한 프로젝트 기준 문서입니다.

본 문서는 `project_brief.md`, `guidelines.md`, `glossary.md`를 참조하며, 특히 현재 기획 단계에서 허용되는 작업과 금지되는 작업을 명확히 하기 위해 작성합니다.

## 2. 프로젝트 개요
- 프로젝트명: 미팅 에이전트
- 프로젝트 목적:
  - Slack 하나로 외부 미팅의 준비(브리핑), 기록(회의록), 후속 처리(Trello·Contacts·내부 미팅 패키지)를 자동화하는 1인용 미팅 라이프사이클 에이전트를 기획·설계하기 위한 기준을 정리합니다.
- 현재 단계: 기획
- 관련 기준 문서:
  - `project_brief.md`

## 3. 현재 단계의 목표
- 이번 단계에서 달성해야 하는 목표:
  - 외부 미팅 라이프사이클 에이전트의 요구사항과 설계 범위를 문서 기준으로 정리합니다.
  - 이후 작업 채팅에서 일관되게 사용할 수 있는 기준 문서를 확정 상태로 만듭니다.
- 이번 단계의 종료 조건:
  - 프로젝트 목적, 사용자, 핵심 흐름, 범위, 제약, 금지사항, 리스크, 미정사항이 문서로 정리되어 있습니다.
  - 구현 단계로 넘어가지 않고도 다음 작업을 위한 기준으로 사용할 수 있습니다.
- 이번 단계에서 만들 산출물:
  - `project_brief.md`
  - `harness.md`
  - 요구사항 + 설계 통합 문서
  - 필요 시 Executive Summary 포함 문서

## 4. 허용 작업
- 현재 단계에서 AI가 수행해도 되는 작업:
  - intake 내용을 구조화합니다.
  - 프로젝트 배경, 목적, 범위, 사용자, 핵심 흐름을 문서화합니다.
  - 요구사항과 설계 항목을 상위 수준에서 통합 정리합니다.
  - 리스크, 미정사항, 가정, 의존성을 별도 항목으로 드러냅니다.
  - 사용자 확인용 비교안, 선택지, 문서 구조안을 제시합니다.
- 초안 작성 가능 범위:
  - `project_brief.md`
  - `harness.md`
  - 요구사항 + 설계 통합 문서
  - Executive Summary
  - 핵심 업무 흐름의 텍스트 기반 설명
- 구조화 가능 범위:
  - 미팅 준비 / 기록 / 후속 처리 / 내부 공유 흐름 분해
  - 단계별 입력/출력 정의
  - 시스템/도구별 역할 구분
  - 승인 포인트 및 미정사항 정리
- 검토 및 정리 가능 범위:
  - 용어 정리
  - 범위 정리
  - 리스크 정리
  - 성공 기준 정리
  - 산출물 구조 정리

## 5. 금지 작업
- 현재 단계에서 AI가 수행하면 안 되는 작업:
  - 구현
  - 코드 생성
  - 다이어그램 생성
  - 실제 연동 상세 설계 확정
  - 운영 배포 상세 설계 확정
- 승인 없이 하면 안 되는 작업:
  - 문서 구조를 크게 변경하는 작업
  - 범위를 확장하는 작업
  - 특정 도구/플랫폼을 확정 전제로 설계를 고정하는 작업
  - 사용자 확인 없이 문서를 확정 상태로 변경하는 작업
- 범위를 벗어나는 작업:
  - 기획 범위를 넘어서는 개발 태스크 분해
  - 상세 API 명세 작성
  - 데이터베이스 스키마 작성
  - UI 시안, 와이어프레임, 플로우차트 작성
- 아직 다루지 않을 작업:
  - 구현 코드
  - 다이어그램
  - 세부 개발 일정
  - 배포/운영 플레이북
  - 상세 테스트 케이스

## 6. 산출물 기준
- 이번 단계의 주요 산출물:
  - `project_brief.md`
  - `harness.md`
  - 요구사항 + 설계 통합 문서
- 산출물 형식:
  - Markdown (`.md`)
- 반드시 포함해야 할 항목:
  - Executive Summary 또는 이에 준하는 상위 요약
  - 프로젝트 목적 및 문제 정의
  - 대상 사용자
  - 핵심 업무 흐름
  - 범위 / 제외 범위
  - 제약사항
  - 리스크 / 미정사항
  - 승인 포인트
- 아직 포함하지 않아야 할 항목:
  - 구현 코드
  - 다이어그램
  - 상세 개발 태스크
  - 확정되지 않은 UI 시안
  - 실제 연동 세부 명세

## 7. 작업 경계
- 이번 프로젝트에서 다루는 범위:
  - 외부 미팅 라이프사이클을 준비, 기록, 후속 처리 관점에서 문서화하는 범위
  - Slack 단일 인터페이스 전제를 기준으로 상위 요구사항과 설계 방향을 정리하는 범위
- 제외 범위:
  - 구현
  - 실제 시스템 연동
  - 운영 자동화 배포
  - UI/UX 상세 설계
- 다음 단계로 넘길 항목:
  - 연동 구조 상세
  - 데이터 저장 방식
  - 상태 전이 및 예외 처리 정의
  - 개발 태스크 및 구현 계획
- 별도 확인이 필요한 항목:
  - 내부 미팅 패키지의 정확한 구성
  - Contacts 반영 범위
  - Trello 후속 처리의 단위와 수준
  - 자동화 트리거 방식

## 8. 제약사항
- 인터페이스 제약:
  - Slack 하나를 주 인터페이스로 가정합니다.
- 시간 제약:
  - TBD
- 비용/예산 제약:
  - TBD
- 인력/리소스 제약:
  - 1인용 에이전트 사용 시나리오를 기준으로 합니다.
- 기술/정책 제약:
  - 현재 단계는 기획 단계이므로 구현 세부를 고정하지 않습니다.
  - 특정 시스템 연동을 전제로 전체 구조를 잠그지 않습니다.

## 9. 승인 및 확인 포인트
- 사용자 확인 후 진행해야 하는 항목:
  - 전체 구조 확정
  - 산출물 범위 확정
  - 내부 미팅 패키지 정의 확정
  - 연동 대상 시스템 범위 확정
- 구조 변경 전 확인이 필요한 항목:
  - 문서 섹션 체계 변경
  - 범위 추가
  - 산출물 추가
- 다음 단계로 넘어가기 전 확인할 항목:
  - 현재 단계 산출물 검토 완료 여부
  - 미정사항 중 필수 결정 항목의 정리 여부
  - 구현 단계 전환 승인 여부
- 미정 상태로 둘 수 있는 항목:
  - 성공 지표의 수치값
  - 세부 연동 방식
  - 보조 사용자 정의
  - 세부 입력 트리거 방식

## 10. 리스크 및 미정사항
- 현재 인지한 리스크:
  - 자동화 범위를 넓게 잡을 경우 기획 단계 문서가 과도하게 비대해질 수 있습니다.
  - 여러 도구를 동시에 포함하면 상위 설계와 구현 설계가 혼재될 수 있습니다.
  - “1인용” 전제와 “후속 처리 자동화” 기대 수준 간 간극이 생길 수 있습니다.
- 미정 사항:
  - 성공 기준의 정량화 수준
  - 내부 미팅 패키지 정의
  - 각 도구 연동의 최소 요구 수준
- 가정:
  - Slack이 유일한 주 인터페이스입니다.
  - 주요 사용자는 1인 실무 담당자입니다.
  - 현재 단계 산출물은 문서 중심입니다.
- 의존성:
  - `project_brief.md`
  - `guidelines.md`
  - `glossary.md`
  - 이후 작성될 요구사항 + 설계 통합 문서

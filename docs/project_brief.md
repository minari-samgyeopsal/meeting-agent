# 프로젝트 브리프

## 1. 문서 목적
이 문서는 미팅 에이전트 프로젝트의 배경, 목적, 범위, 대상 사용자, 현재 단계, 기대 산출물, 성공 기준, 주요 제약을 정리하기 위한 프로젝트 기준 문서입니다.

이 문서는 `project_intake_template.md`를 바탕으로 작성되었으며, 이후 `harness.md` 및 작업 문서의 기준 문서로 사용합니다.

## 2. 프로젝트 개요
- 프로젝트명: 미팅 에이전트
- 한 줄 설명: Slack 하나로 외부 미팅의 준비(브리핑), 기록(회의록), 후속 처리(Trello·Contacts·내부 미팅 패키지)를 자동화하는 1인용 미팅 라이프사이클 에이전트
- 현재 단계: 기획
- 프로젝트 요약:
  - 본 프로젝트는 외부 미팅 전후에 반복적으로 발생하는 준비, 기록, 후속 처리 업무를 하나의 흐름으로 정리하기 위한 1인용 에이전트 프로젝트입니다.
  - 핵심 목적은 외부 미팅 관련 작업을 분절된 수작업이 아니라 하나의 라이프사이클로 연결하는 것입니다.
  - 이번 단계에서는 구현이 아니라 요구사항과 설계 기준을 정리하는 데 집중합니다.
  - 구현, 코드, 다이어그램, 실제 연동 상세는 본 단계 범위에 포함하지 않습니다.

## 3. 배경 및 문제 정의
- 이 프로젝트를 하게 된 배경:
  - 외부 미팅 관련 업무가 미팅 전 준비, 미팅 중 기록, 미팅 후 후속 처리로 나뉘어 있으며, 현재는 이를 하나의 흐름으로 일관되게 관리하기 어렵습니다.
- 해결하려는 핵심 문제:
  - 외부 미팅 라이프사이클 전반의 반복 수작업을 줄이고, 준비 누락·기록 누락·후속 처리 누락을 최소화하는 것입니다.
- 현재 방식의 페인포인트:
  - 외부 미팅 상대방 정보를 매번 수작업으로 검색·정리해야 합니다.
  - 미팅 후 회의록 작성에 시간이 많이 소요됩니다.
  - 외부 미팅 직후 내부 미팅이 이어질 때 별도 정리 없이 대응해야 하는 상황이 반복됩니다.
  - 같은 업체와 반복 미팅할 때 과거 맥락이 체계적으로 누적 관리되지 않습니다.
  - Trello, Contacts, 내부 공유용 패키지 등 후속 처리가 각각 분리된 수작업으로 남아 있습니다.
- 지금 이 프로젝트가 필요한 이유:
  - 현재 업무 방식은 반복 비용이 높고, 사람의 기억과 수동 정리에 과도하게 의존합니다.
  - 준비, 기록, 후속 조치가 하나의 흐름으로 연결되지 않으면 미팅 품질과 후속 실행력이 일관되게 유지되기 어렵습니다.

## 4. 프로젝트 목표
- 이번 프로젝트의 목표:
  - 외부 미팅의 준비, 기록, 후속 처리 흐름을 하나의 1인용 미팅 라이프사이클 에이전트 관점에서 정의하고 기준 문서화합니다.
- 이번 단계의 목표:
  - 요구사항과 설계 관점을 통합한 기준 문서 작성을 위한 프로젝트 기준을 확정 가능한 수준으로 정리합니다.
- 이번 단계에서 도달해야 하는 상태:
  - 프로젝트의 목적, 범위, 사용자, 핵심 흐름, 제약, 금지사항, 리스크, 미정사항이 문서 기준으로 정리되어 있어야 합니다.
- 이번 단계에서 하지 않는 것:
  - 구현
  - 코드 생성
  - 다이어그램 생성
  - 실제 연동 상세 설계 확정
  - 사용자 확인 전 문서 확정

## 5. 대상 사용자 및 사용 상황
- 주요 사용자:
  - 1인 실무 담당자
- 보조 사용자(있다면):
  - 미정
- 사용 상황:
  - 외부 미팅 전 상대방 및 아젠다 브리핑이 필요한 상황
  - 외부 미팅 직후 회의록을 빠르게 정리해야 하는 상황
  - 후속 액션을 Trello, Contacts, 내부 공유용 문서 또는 내부 미팅 패키지로 연결해야 하는 상황
- 핵심 업무 흐름 또는 사용 시나리오:
  - Slack에서 미팅 관련 요청을 시작합니다.
  - 외부 미팅 전 브리핑 자료를 준비합니다.
  - 미팅 후 회의록을 정리합니다.
  - 후속 조치 항목을 정리하고 필요한 시스템 또는 내부 패키지로 연결합니다.
  - 같은 상대 또는 조직과의 후속 미팅에 재사용 가능한 맥락을 남깁니다.

## 6. 이번 단계의 산출물
- 이번 단계의 주요 산출물:
  - 요구사항 + 설계 통합 문서
  - `project_brief.md`
  - `harness.md`
- 산출물 형식:
  - Markdown (`.md`)
- 반드시 포함해야 할 항목:
  - 프로젝트 배경
  - 문제 정의
  - 프로젝트 목표
  - 대상 사용자
  - 핵심 업무 흐름
  - 범위 및 제외 범위
  - 제약사항
  - 리스크 및 미정사항
- 필요 시 포함할 보조 산출물:
  - Executive Summary
  - 리스크 / 미정사항 정리
  - 용어 정의 초안

## 7. 성공 기준
- 이번 단계가 완료되었다고 볼 수 있는 기준:
  - 미팅 에이전트 프로젝트의 목적, 범위, 주요 흐름, 제약, 금지사항이 문서로 정리되어 이후 작업의 기준으로 사용할 수 있어야 합니다.
- 산출물이 충족해야 하는 상태:
  - 저장 가능한 형식이어야 하며, 대화형 메모가 아니라 바로 기준 문서로 활용 가능해야 합니다.
  - 구현 단계로 넘어가기 전 상위 기준 문서로 사용 가능한 수준이어야 합니다.
- 사용자 관점에서 성공이라고 볼 수 있는 조건:
  - 추가 설명 없이도 이후 작업 채팅에서 동일한 방향으로 요구사항/설계 작업을 이어갈 수 있어야 합니다.
  - 현재 단계에서 무엇을 해도 되고 무엇을 하면 안 되는지가 명확해야 합니다.

## 8. 범위
- 이번에 포함할 범위:
  - 외부 미팅 라이프사이클의 상위 구조 정의
  - Slack 중심 인터페이스 가정
  - 브리핑, 회의록, 후속 처리, 내부 미팅 패키지의 기능 범위 정리
  - 요구사항과 설계 관점의 문서화
- 제외할 범위:
  - 실제 구현
  - 코드 작성
  - 다이어그램 작성
  - 실제 API 연동 상세 확정
  - 운영 배포 방안 확정
- 나중에 검토할 범위:
  - Trello·Contacts 등 연동 세부 방식
  - 데이터 저장 구조
  - 상태 관리 방식
  - 예외 처리 흐름
  - 권한 및 보안 세부 정책
- 절대 하면 안 되는 것:
  - 승인 전 구현
  - 코드 생성
  - 다이어그램 생성
  - 사용자 확인 전 문서 확정

## 10. 미정 사항 및 확인 필요 항목
- 아직 미정인 사항:
  - 보조 사용자 존재 여부
  - 성공 기준의 정량 지표
  - 시간/예산 제약
  - Trello·Contacts·내부 미팅 패키지의 세부 산출 형식
  - 미팅 입력 방식(수동 입력 / 캘린더 트리거 / 기타)
- 추가 확인이 필요한 항목:
  - 내부 미팅 패키지의 정확한 구성 범위
  - 후속 처리 자동화의 최소 단위
  - 어떤 시스템까지 이번 설계 범위에 포함할지
- 현재 가정하고 있는 내용:
  - Slack이 단일 인터페이스입니다.
  - 주요 사용자는 1인 실무 담당자입니다.
  - 본 단계의 산출물은 문서 기준 정리에 한정됩니다.
- 주요 리스크:
  - 범위가 넓어질 경우 기획 단계에서 구현 상세로 과도하게 이동할 위험이 있습니다.
  - 연동 대상 시스템이 늘어나면서 문서 범위가 불필요하게 비대해질 위험이 있습니다.
  - “자동화” 범위를 명확히 정의하지 않으면 기대 수준이 과도하게 커질 위험이 있습니다.

# 용어집

## 1. 목적
이 문서는 에이전트 프로젝트 운영과 회사 내 AI 협업에서 사용하는 핵심 용어를 일관되게 정의하기 위한 문서입니다.
같은 단어를 프로젝트마다 다르게 해석하거나, 일반 의미와 프로젝트 특화 의미가 혼재되는 것을 줄이기 위해 사용합니다.

이 문서는 이미 합의된 용어를 저장하는 문서입니다.
모호한 용어가 등장했을 때 어떻게 확인하고 처리할지는 `guidelines.md`의 용어 해석 원칙을 따릅니다.

---

## 2. 시스템 운영 용어

### 공통 가이드라인 (Guidelines)
여러 프로젝트에 공통 적용되는 AI 협업 원칙입니다.
프로젝트별 제약이 아니라 상위 공통 문서입니다.

파일명:
`guidelines.md`

### 프로젝트 Intake
새 프로젝트 시작 시 최소한의 정보를 수집하고 정리하는 절차입니다.
이 절차를 통해 프로젝트 기준 문서를 생성합니다.

### 프로젝트 Intake 템플릿
프로젝트 시작 시 필요한 정보를 구조적으로 받기 위한 입력 양식입니다.

파일명:
`project_intake_template.md`

### 프로젝트 브리프 (Project Brief)
프로젝트의 배경, 목적, 범위, 대상 사용자, 현재 단계, 기대 산출물, 성공 기준, 주요 제약을 정리한 기준 문서입니다.

파일명:
`project_brief.md`

### 하네스 (Harness)
해당 프로젝트에서 AI가 따라야 할 작업 경계, 허용 작업, 금지 작업, 산출물 기준, 제약사항, 승인 포인트, 리스크, 미정 사항을 정리한 프로젝트 기준 문서입니다.

파일명:
`harness.md`

### 하네스 템플릿
프로젝트별 `harness.md`를 만들기 위한 템플릿 문서입니다.

파일명:
`harness_template.md`

### 스캐폴드 (Scaffold)
산출물의 기본 구조, 목차, 섹션 구성을 정리한 문서 또는 구조입니다.
AI가 임의로 문서 구조를 흔들지 않도록 기본 뼈대를 고정하는 역할을 합니다.

파일명 예시:
`scaffold.md`

### 루브릭 (Rubric)
산출물의 품질을 평가하기 위한 기준입니다.
무엇이 좋은 결과물인지 판단하기 위한 평가 항목을 정의합니다.

파일명 예시:
`rubric.md`

### DoD (Definition of Done)
산출물이 완료되었다고 판단하기 위한 완료 기준입니다.
필수 요건 충족 여부를 확인하는 문서입니다.

파일명 예시:
`dod.md`

### 프로젝트 기준 문서
해당 프로젝트의 기준이 되는 핵심 문서입니다.
일반적으로 아래 두 문서를 의미합니다.

- `project_brief.md`
- `harness.md`

이후 작업은 이 문서를 기준으로 진행합니다.

### 작업 문서
프로젝트 수행 중 생성되는 문서입니다.

예:
- 요구사항 문서
- 설계 문서
- `scaffold.md`
- `rubric.md`
- `dod.md`
- 세션 상태 문서
- 기타 실행 문서

### Intake 채팅
새 프로젝트의 정보를 수집하고 `project_brief.md`, `harness.md`를 만들기 위한 초기 설정용 채팅입니다.
실제 산출물 작성 채팅과 구분합니다.

### 작업 채팅
프로젝트 내에서 실제 문서 작성, 설계, 검증 등을 수행하는 대화 단위입니다.

### 회고 (Retrospective)
프로젝트 종료 후 무엇이 잘 되었고, 무엇이 부족했고, 다음 프로젝트에 무엇을 반영할지를 정리하는 문서입니다.

파일명 예시:
`retrospective.md`

### Lessons
특정 프로젝트를 수행하며 얻은 학습사항입니다.
반드시 모든 프로젝트에 공통 적용되는 것은 아니며, 경험 기반 교훈에 가깝습니다.

파일명 예시:
`lessons_candidate.md`

### Patterns
다른 유사 프로젝트에도 반복 적용할 수 있는 구조, 방식, 절차, 문서 패턴입니다.
일회성 예외가 아니라 재사용 가능성이 있어야 합니다.

파일명 예시:
`pattern_candidate.md`

### 상위 공통 문서
여러 프로젝트에서 공통 참조하는 문서입니다.

예:
- `guidelines.md`
- `glossary.md`
- `project_intake_template.md`
- `harness_template.md`

## 3. 회사 / 도메인 용어

이 섹션은 회사에서 반복적으로 사용하는 업무, 조직, 제품, 파트너십, 운영 관련 용어를 정리하기 위한 영역입니다.
초기에는 비워둘 수 있으며, 실제 프로젝트를 진행하면서 반복 등장하는 용어를 검토 후 추가합니다.

### 예시로 들어갈 수 있는 항목
- 조직명
- 제품명
- 내부 약어
- 자주 쓰는 사업개발 용어
- 회사 고유의 의사결정 표현
- 파트너십 단계 정의
- PoC / Pilot / Launch의 내부 정의 차이
- KPI 관련 내부 표현

※ 위 목록은 예시이며, 확정된 용어만 반영합니다.

## 4. 용어 업데이트 원칙

### 4.1 glossary에는 합의된 용어만 반영한다
glossary는 이미 정의가 합의된 용어를 저장하는 문서입니다.
프로젝트 중 임시로 등장한 표현을 바로 반영하지 않습니다.

### 4.2 새 용어는 먼저 프로젝트 안에서 확인한다
glossary에 없는 용어이지만 프로젝트상 중요도가 높은 경우, 먼저 해당 프로젝트 문맥에서 의미를 확인하고 임시 정의로 사용합니다.

### 4.3 반복 사용 가능성이 검증된 경우에만 승격한다
새 용어가 여러 프로젝트에서 반복 사용되거나 회사 공통 맥락에서 유효하다고 판단되는 경우에만 glossary 반영 후보로 올립니다.

### 4.4 일반 의미와 회사 의미가 다르면 회사 의미를 명시한다
동일한 단어가 일반 의미와 회사 내 의미를 다르게 가질 수 있는 경우, glossary에는 회사에서 사용하는 의미를 명확히 적습니다.

### 4.5 업데이트는 검토 후 반영한다
glossary는 자동으로 수정하지 않습니다.
프로젝트 수행 중 나온 후보 용어는 회고 또는 별도 검토를 거쳐 반영합니다.

## 5. 관리 메모
- 시스템 운영 용어와 회사/도메인 용어를 구분해 관리합니다.
- 이 문서는 프로젝트마다 새로 만들지 않고 상위 공통 문서로 유지합니다.
- 반복적으로 충돌하거나 해석 차이가 발생한 용어를 우선 보강합니다.
